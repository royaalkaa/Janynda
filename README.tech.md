# Краткое техническое описание Janynda

## 1. Назначение проекта

Janynda — веб-приложение для семейного ухода за пожилыми родственниками. Система объединяет бытовое планирование, ввод показателей здоровья, геолокационный контроль, уведомления и голосового помощника в одном интерфейсе как для пожилого пользователя, так и для наблюдающих членов семьи.

## 2. Архитектура

Проект реализован как модульный монолит на Django.

Бэкенд:

- Django 5.1
- PostgreSQL как основная база данных
- Redis для кеша и как брокер/хранилище результатов Celery
- Celery + django-celery-beat для фоновых задач

Фронтенд:

- шаблоны Django
- HTMX для частичных обновлений без тяжёлой SPA-архитектуры
- Bootstrap 5 как базовый UI-слой
- Chart.js для графиков

Модель развёртывания:

- контейнер `web` для Django
- контейнер `db` для PostgreSQL
- контейнер `redis` для Redis
- контейнер `celery` для фонового выполнения задач
- контейнер `celery-beat` для периодических задач

## 3. Основные доменные модули

`accounts`

- собственная модель `User` с ролями `observer`, `subject` или `both`
- профиль пользователя
- вход по magic-link для упрощённого доступа пожилого пользователя

`family`

- семейные группы
- связи observer -> subject
- приглашения и модель доступа

`care`

- элементы плана дня
- повторяющиеся задачи
- полезные места
- геолокация
- безопасные зоны
- сводки по wearable-данным

`health`

- универсальное хранение метрик через `MetricRecord`
- пороги по субъекту
- расчёт серьёзности: `normal`, `warning`, `critical`
- цели по здоровью

`notifications`

- центр уведомлений внутри приложения
- настройки уведомлений
- модель важности и категорий

`ai_assistant`

- история AI-чата
- кешированные AI-комментарии
- журнал голосовых команд
- fallback-логика при отсутствии OpenAI

`dashboard`

- дашборды subject и observer
- последние снимки метрик
- агрегированные данные для графиков

## 4. Ключевая модель данных

Основные сущности, вокруг которых построена система:

- `User` и `UserProfile`
- `FamilyGroup` и `FamilyMembership`
- `DailyPlanItem`
- `MetricRecord` и `ThresholdConfig`
- `Notification`
- `AIComment`, `AIConversation`, `AIMessage`, `VoiceCommandLog`
- погодный кеш и сущности, связанные с геолокацией

Важное архитектурное решение: медицинские измерения хранятся в универсальной структуре. Одна модель обслуживает разные типы метрик, а значения лежат в JSON. Это уменьшает количество специализированных таблиц и упрощает добавление новых типов показателей.

## 5. Фоновая обработка

Периодические задачи выполняются через Celery и Celery Beat. Основные процессы:

- генерация повторяющихся задач плана дня
- отправка напоминаний по задачам
- проверка отсутствия пользователя по геолокации
- проверка целей по wearable-метрикам
- реакция на события новых геопозиций

Такой подход отделяет отображение интерфейса от асинхронных проверок и напоминаний.

## 6. Внешние интеграции

Проект умеет интегрироваться с:

- OpenWeatherMap для погоды и качества воздуха
- OpenAI для коротких ответов ассистента по здоровью
- браузерными Speech API для голосового взаимодействия
- Yandex Maps для отображения карты в браузере

Важный момент: приложение спроектировано с частичными fallback-сценариями:

- без `OPENAI_API_KEY` ассистент использует локальные fallback-ответы по правилам
- без внешних API-ключей основное приложение всё равно запускается, а базовые CRUD-сценарии остаются рабочими

## 7. Модель доступа и безопасность

Система разделяет доступ по ролям:

- `subject` видит свой дашборд и может вводить данные
- `observer` наблюдает за родственниками через семейные связи
- magic links упрощают вход для пожилых пользователей, которым неудобен полный парольный сценарий

Критичные production-настройки вынесены в `config.settings.prod`, а разработка и тесты используют отдельные settings-модули.

## 8. Что стоит проговорить на защите

Ключевые акценты для защиты:

- это не статичный прототип, а полноценная full-stack-система с БД, фоновыми воркерами и интеграциями
- Django выбран из-за скорости разработки, встроенного ORM, админки, аутентификации, форм и шаблонов
- Celery нужен, потому что напоминания и проверки состояния не должны блокировать HTTP-запросы
- Redis используется как инфраструктурный слой для кеша и очереди задач
- AI-часть опциональна и изолирована, поэтому система остаётся рабочей даже без платного API
- HTMX выбран как компромисс: интерфейс остаётся динамичным без усложнения архитектуры SPA
- текущую архитектуру легко масштабировать: можно добавлять новые задачи, интеграции, метрики и клиентов поверх существующего бэкенда
